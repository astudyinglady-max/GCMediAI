# GitHub Projects 칸반 보드 구성 가이드

> **프로젝트**: GCMediAI 회사소개 홈페이지 리뉴얼
> **작성일**: 2026-03-16

---

## 보드 생성 방법

1. GitHub 레포지토리 상단 탭에서 **Projects** 클릭
2. **New project** 버튼 클릭
3. 템플릿 선택 화면에서 **Board** 선택
4. 프로젝트 이름 입력 (예: `GCMediAI 홈페이지 리뉴얼`) → **Create project**

## 컬럼 구성

기본 생성된 컬럼을 삭제하고 아래 5개로 구성한다.

| 순서 | 컬럼명 | 용도 |
|------|--------|------|
| 1 | **Backlog** | 작업이 확정되었으나 아직 시작 전인 이슈 |
| 2 | **In Progress** | 현재 진행 중인 작업 |
| 3 | **In Review** | PR이 올라가 코드 리뷰 대기 또는 진행 중인 상태 |
| 4 | **QC** | 머지 후 Vercel Preview에서 기획·디자인 검수 중인 상태 |
| 5 | **Done** | 검수 완료 및 develop 반영 완료 |

### 컬럼별 상세 설명

#### Backlog
- 스프린트 계획 시 이슈를 생성해 이 컬럼에 쌓아둔다.
- 우선순위 순서로 위에서 아래로 정렬한다. (위쪽 = 높은 우선순위)
- 이슈 생성 시 담당자(Assignee), 레이블, 마일스톤을 함께 설정한다.

#### In Progress
- 담당자가 작업을 시작하면 직접 이 컬럼으로 이동한다.
- **WIP(Work In Progress) 제한 권장**: 1인당 동시 진행 이슈 2개 이하.
- 작업 브랜치(`feat/...`)를 생성하고 이슈 번호를 브랜치명 또는 커밋에 연결한다.

#### In Review
- PR을 `develop`으로 올린 후 이 컬럼으로 이동한다.
- PR 링크를 이슈 코멘트에 첨부한다.
- 리뷰어가 Approve하면 담당자가 머지 후 QC로 이동한다.

#### QC (Quality Check)
- `develop` 머지 후 Vercel Preview URL에서 기획·디자인 담당자가 검수한다.
- 검수 항목: 화면 레이아웃, 반응형, 텍스트 오탈자, 디자인 시안 일치 여부.
- 수정 사항 발견 시 이슈 코멘트에 피드백을 남기고 In Progress로 되돌린다.
- 이상 없으면 Done으로 이동한다.

#### Done
- 검수 완료 및 `develop` 반영이 확정된 이슈.
- 스프린트 회고 시 이 컬럼을 기준으로 완료 작업량을 집계한다.
- 스프린트 종료 후 `develop → main` 배포 시 해당 스프린트 이슈가 모두 Done에 있는지 확인한다.

## 컬럼 생성 방법

1. 보드 오른쪽 끝 **+ Add column** 클릭
2. 컬럼 이름 입력 후 저장
3. 컬럼을 드래그해 위 순서대로 배치

## 이슈 → 보드 자동화 설정

GitHub Projects의 **Workflows** 기능으로 이슈/PR 상태 변경을 자동화한다.

| 트리거 | 자동 이동 | 설정 방법 |
|--------|----------|----------|
| 이슈 생성 | → Backlog | Workflows → `Item added to project` → Status: Backlog |
| PR 오픈 | → In Review | Workflows → `Pull request opened` → Status: In Review |
| PR 머지 | → QC | Workflows → `Pull request merged` → Status: QC |
| 이슈 Close | → Done | Workflows → `Item closed` → Status: Done |

### 자동화 설정 방법

1. 프로젝트 우측 상단 **...** → **Workflows** 클릭
2. 각 워크플로 항목 클릭 → **Edit** → 조건·이동 컬럼 설정 → **Save and turn on**

## 레이블 구성 권장

이슈 분류를 위해 아래 레이블을 레포지토리에 추가한다.

| 레이블 | 색상 | 용도 |
|--------|------|------|
| `feat` | `#0075ca` | 기능 개발 |
| `fix` | `#d73a4a` | 버그 수정 |
| `docs` | `#e4e669` | 문서 작업 |
| `design` | `#f9a8d4` | 디자인 산출물 |
| `planning` | `#c5def5` | 기획 산출물 |
| `chore` | `#ededed` | 설정·빌드 |
| `blocked` | `#b60205` | 외부 의존으로 진행 불가 |

### 레이블 추가 방법

레포지토리 → **Issues** 탭 → **Labels** → **New label**

## 마일스톤 구성 권장

스프린트 단위로 마일스톤을 생성해 이슈와 연결하면 진척도를 퍼센트로 확인할 수 있다.

| 마일스톤 | 내용 |
|----------|------|
| `스프린트 1 — 기획·설계` | 요구사항, 유저스토리, 기술스택, 브랜치 전략 확정 |
| `스프린트 2 — 디자인·설계` | 디자인 시안, 컴포넌트 설계 |
| `스프린트 3 — 개발 MVP` | 페이지 구현, 반응형, SEO |
| `스프린트 4 — 폴리싱·배포` | 애니메이션, 성능 최적화, 배포 |

### 마일스톤 추가 방법

레포지토리 → **Issues** 탭 → **Milestones** → **New milestone**
