## 스프린트 1 — 기획·설계 (2026-03-16 ~)

### 2026-03-16

#### 확정된 요구사항

**페이지 구성 (5개)**
- 메인 `/`, 회사소개 `/about`, 사업분야 `/business`, IR `/ir`, 문의하기 `/contact`
- 추가 확정: 개인정보처리방침 `/privacy`, 404·500 에러 페이지

**Must Have 핵심 항목**
- 메인: 히어로("Beyond EMR, Toward Medical OS"), AI 수치(25,000처+·2억 건+·Medical LLM), 사업분야 요약 카드(4개 영역), 뉴스 미리보기(3~5건), IR 정적 미리보기, CTA
- 사업분야: 4개 영역(메디컬 인프라·데이터·마케팅·플랫폼·유통·의료기기) / 5개 서비스(의사랑·유팜·UBIST·똑닥·미소몰)
- IR: 정적 재무정보·공시·주주공고·NEWS. 주가정보는 Should Have로 분류(MVP 이후 추가)
- 문의하기: 문의 유형 선택(사업제휴·채용·언론·기타) Must Have로 확정. MVP 제출 처리는 Server Action 서버 로그 기록, 이메일 발송은 Should Have
- 공통: 푸터에 사업자등록번호·개인정보처리방침 링크 포함

**성능 목표**
- Lighthouse 80점 이상 + Core Web Vitals (LCP < 2.5s, CLS < 0.1, INP < 200ms) Must Have로 통합

**접근성**
- 키보드 내비게이션, 이미지 alt, 폼 aria 속성 → Must Have
- WCAG 2.1 AA 전면 준수 → Should Have (공공기관 대상 서비스 여부 확인 후 상향 검토)

#### 주요 논의 사항

**1. 사업분야 "4개 영역 vs 5개 서비스" 표기 혼용 문제**
- 결론: 전 문서에서 "4개 영역(5개 서비스)"으로 통일
  - 메디컬 인프라(의사랑·유팜) / 데이터·마케팅(UBIST) / 플랫폼(똑닥) / 유통·의료기기(미소몰)
- 배경: 요구사항·유저스토리·MVP 범위 섹션에서 숫자가 혼용되어 개발 착수 전 혼동 유발 가능성 있었음

**2. 문의 폼 MVP 제출 처리 방식**
- 결론: 이메일 발송은 MVP 제외. Server Action에서 서버 로그 기록 → 제출 성공 메시지 표시
- 배경: 이메일 서버(Resend) 설정에 외부 계정 발급이 필요하여 MVP 착수를 지연시킬 수 있음
- 후속: 스프린트 2~3에서 Resend 연동 예정

**3. TanStack Query MVP 제외**
- 결론: MVP 기술 스택에서 제외, 확장 단계로 이동
- 배경: "MVP에서는 Server Components fetch로 충분하다"고 기술스택 문서 자체에서 인정했으면서 스택에 포함시킨 모순을 크로스 리뷰에서 발견
- 후속: IR 실시간 데이터·CMS 연동 시 도입

**4. Lighthouse Must Have vs CWV Should Have 모순**
- 결론: CWV 항목을 Must Have로 통합. "Lighthouse 80점 이상(CWV 포함)" 단일 목표로 정리
- 배경: LCP < 2.5s를 달성 못하면 Lighthouse 80점도 불가능하므로 사실상 동일 목표였음

**5. 접근성 우선순위**
- 결론: 기본 접근성(키보드·alt·폼 aria·색상 대비)은 Must Have로 분리. WCAG 2.1 AA는 Should Have 유지
- 배경: GC메디아이가 공공기관 대상 서비스를 운영하는 경우 법적 의무 대상일 수 있으므로 확인 필요

**6. 개인정보처리방침 페이지 누락 발견**
- 결론: `/privacy` 페이지 Must Have로 추가. 푸터 링크 Must Have에 포함
- 배경: 유저스토리 AC에서는 언급했으나 요구사항·페이지 구성에서 누락되어 있었음

**7. Google Analytics 우선순위**
- 결론: Nice to Have → Should Have로 상향. Vercel Analytics(무료 내장)를 MVP 대안으로 먼저 적용
- 배경: "도메인 확정 후"라는 이유로 Nice to Have였으나, GA 스크립트 삽입은 도메인 무관. Vercel Analytics는 별도 설정 없이 즉시 사용 가능

## 스프린트 로그 작성 가이드

> 스프린트 진행 중 이 파일을 업데이트하는 방법.

```
## 스프린트 N — 제목 (YYYY-MM-DD ~)

### YYYY-MM-DD

#### 완료된 작업
#### 발생한 이슈 / 결정 사항
#### 다음 단계
```

- **언제 업데이트**: 스프린트 시작일, 주요 결정이 있는 날, 스프린트 종료일
- **누가 업데이트**: 해당 날짜 작업을 주도한 담당자가 PR에 포함하거나 별도 커밋으로 추가
- **커밋 메시지**: `docs: 스프린트 N 진행 기록 추가 (YYYY-MM-DD)`
