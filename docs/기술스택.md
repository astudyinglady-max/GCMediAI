## 1. 프론트엔드

### 1-1. Next.js 15 (App Router)

#### 선택 이유

- **SSR/SSG/ISR 통합 지원**: 회사소개 홈페이지는 대부분 정적 콘텐츠지만, IR 공시 목록이나 뉴스 같은 콘텐츠는 주기적 갱신이 필요하다. Next.js는 페이지별로 렌더링 전략(Static → ISR → SSR)을 선택할 수 있어 유연성이 높다.
- **App Router + Server Components**: React Server Component(RSC)로 서버에서 HTML을 완성해 전송하므로, 초기 로드 속도와 SEO가 모두 개선된다. 요구사항의 Lighthouse 80점 이상 달성에 유리하다.
- **내장 SEO 지원**: `metadata` API로 페이지별 `<title>`, `<meta>`, Open Graph를 타입 안전하게 설정할 수 있다. 별도 라이브러리 없이 요구사항의 SEO 항목을 충족한다.
- **내장 Image 최적화**: `next/image`로 WebP 변환, 지연 로딩, 반응형 srcset을 자동 처리한다.
- **Vercel과의 최적 통합**: 배포 플랫폼 Vercel이 Next.js의 제작사로, 제로 설정 배포 및 Edge Functions 지원이 기본 제공된다.

#### 대안과의 비교

| 기준 | **Next.js 15** | Nuxt 3 | Remix | Astro |
|------|:-----------:|:------:|:-----:|:-----:|
| React 생태계 활용 | ✅ | ❌ (Vue) | ✅ | 부분 |
| SSR + SSG 혼합 | ✅ | ✅ | SSR 중심 | SSG 중심 |
| App Router (RSC) | ✅ | ❌ | ❌ | ❌ |
| Vercel 네이티브 지원 | ✅ | 보통 | 보통 | 보통 |
| 국내 레퍼런스 | ✅ (압도적) | 보통 | 적음 | 적음 |
| 팀 학습 비용 | 낮음 | 높음 (Vue 전환) | 중간 | 중간 |

- **Remix 미채택 이유**: 강제적인 SSR 중심 구조가 정적 콘텐츠 위주의 회사소개 사이트에 과하다. ISR 지원도 Next.js 대비 제한적이다.
- **Astro 미채택 이유**: 정적 사이트에는 최적이나, IR 데이터나 문의 폼 같은 서버 기능이 필요한 페이지에서 Next.js 대비 생산성이 낮다.
- **Nuxt 3 미채택 이유**: Vue 기반으로 기존 React 팀 역량과 불일치한다.

#### 팀 역량 적합성

- React 기반 개발 경험을 가진 팀에게 Next.js는 점진적으로 학습 가능하다.
- App Router가 Pages Router보다 학습 곡선이 있으나, Server Component의 장점(번들 크기 감소, SEO)이 이를 상쇄한다.
- 국내 커뮤니티와 레퍼런스가 풍부해 문제 해결이 빠르다.

### 1-2. React 19

#### 선택 이유

- Next.js 15의 기본 React 버전이 19다. 별도 선택이 아닌 Next.js 15를 선택함에 따른 자연스러운 의존성이다.
- **React 19 주요 개선점**:
  - `use()` hook: 비동기 데이터 처리가 단순해짐
  - `useActionState` / `useFormStatus`: Server Actions와 연동되는 폼 상태 관리 내장
  - 자동 메모이제이션 (React Compiler 지원): 불필요한 `useMemo`/`useCallback` 감소

#### 대안과의 비교

React 18과의 비교:

| 기준 | React 19 | React 18 |
|------|:--------:|:--------:|
| Server Actions 폼 연동 | ✅ (내장) | 직접 구현 필요 |
| `use()` hook | ✅ | ❌ |
| 자동 메모이제이션 | ✅ (실험적) | ❌ |
| 안정성 | ✅ (정식 릴리스) | ✅ |

#### 팀 역량 적합성

- React 18 이상 경험이 있다면 19로의 전환 비용이 낮다.
- 새로운 hook은 점진적으로 적용 가능하며, 기존 React 패턴이 그대로 동작한다.

### 1-3. TypeScript

#### 선택 이유

- **타입 안전성**: IR 데이터, 뉴스 데이터 등 외부에서 오는 데이터 구조를 타입으로 명세하면 런타임 오류를 컴파일 타임에 잡을 수 있다.
- **IDE 자동완성**: 컴포넌트 props, API 응답 타입이 명세되면 개발 속도가 크게 향상된다.
- **리팩토링 안전성**: 추후 컴포넌트 구조 변경이나 API 변경 시 영향 범위를 타입 에러로 즉시 파악한다.
- **팀 협업**: 여러 명이 동일 코드베이스를 다룰 때 타입이 인터페이스 계약 역할을 한다.

#### 대안과의 비교

| 기준 | **TypeScript** | JavaScript (JSDoc) | JavaScript (plain) |
|------|:------------:|:-----------------:|:-----------------:|
| 타입 안전성 | ✅ (완전) | 부분적 | ❌ |
| IDE 지원 | ✅ | 보통 | 낮음 |
| 컴파일 타임 에러 감지 | ✅ | 제한적 | ❌ |
| 초기 설정 비용 | 낮음 (Next.js 기본 제공) | 없음 | 없음 |
| 장기 유지보수 | ✅ (우수) | 보통 | 어려움 |

- **JSDoc 미채택 이유**: 대규모 팀이나 복잡한 데이터 구조에서는 TypeScript보다 표현력이 부족하다. Next.js 프로젝트 기본값이 TypeScript인 점도 고려했다.

#### 팀 역량 적합성

- 현대 웹 개발 표준으로 자리잡아 팀 내 경험이 있을 가능성이 높다.
- `any` 타입을 금지하는 컨벤션(`CLAUDE.md` 참고)을 통해 TypeScript의 이점을 최대화한다.

### 1-4. SCSS

#### 선택 이유

- **중첩(Nesting) 문법**: 컴포넌트 스코프 내 스타일을 계층적으로 표현하여 가독성이 높다.
- **변수 및 믹스인**: 브레이크포인트, 색상, 타이포그래피 등 디자인 토큰을 변수로 관리한다.
- **CSS와의 호환성**: 순수 CSS 문법을 그대로 사용할 수 있어 학습 곡선이 낮다.
- **빌드 결과물 예측 가능**: 컴파일된 CSS가 명확하여 디버깅이 쉽다.

#### 대안과의 비교

| 기준 | **SCSS** | Tailwind CSS | CSS Modules | CSS-in-JS (styled-components) |
|------|:-------:|:------------:|:-----------:|:-----------------------------:|
| 러닝 커브 | 낮음 | 중간 (유틸리티 암기) | 낮음 | 중간 |
| 디자인 시스템 관리 | ✅ (변수/믹스인) | ✅ (config) | 제한적 | ✅ |
| SSR 호환성 | ✅ | ✅ | ✅ | 주의 필요 |
| 번들 크기 | 작음 | 작음 (PurgeCSS) | 작음 | 큼 (런타임) |
| 인라인 스타일 없이 작업 | ✅ | 클래스 비대화 | ✅ | ✅ |
| 팀 친숙도 | 높음 | 중간 | 높음 | 중간 |

- **Tailwind 미채택 이유**: 유틸리티 클래스가 HTML을 비대하게 만들고, 커스텀 디자인이 많은 회사소개 사이트에서 `className` 조합이 복잡해진다. SCSS로 컴포넌트 단위 스타일 분리가 더 명확하다.
- **CSS-in-JS 미채택 이유**: styled-components, Emotion 등은 런타임 비용이 있고, Next.js App Router의 RSC와 충돌 가능성이 있다. SSR 환경에서 스타일 hydration 이슈가 발생할 수 있다.
- **CSS Modules 미채택 이유**: 기능적으로 유사하지만 변수, 믹스인 등 SCSS의 추가 기능이 없어 디자인 토큰 관리가 불편하다. 최신 네이티브 CSS(Custom Properties, CSS Nesting)가 상당 부분을 커버하지만, `@use`/`@forward` 모듈 시스템, `@mixin`/`@include`, 조건문·반복문(`@each`, `@for`) 등 SCSS 고유 기능이 복잡한 디자인 시스템 관리에 유리하다. 2026년 시점에서 네이티브 CSS가 빠르게 발전하고 있으나, 브레이크포인트 믹스인·반복 패턴의 편의성은 SCSS가 여전히 우위에 있다.

#### 팀 역량 적합성

- CSS 경험이 있는 개발자라면 SCSS의 추가 문법 학습 비용이 1~2일 수준으로 낮다.
- 컴포넌트별 `.scss` 파일로 관리하므로 코드 오너십이 명확하다.

### 1-5. TanStack Query v5 (확장 단계 도입)

> **MVP에서는 미사용**. MVP의 모든 데이터가 정적 JSON이고, 문의 폼은 Server Actions로 처리하므로 Next.js Server Components의 `fetch`만으로 충분하다. TanStack Query는 IR API 연동 또는 뉴스 CMS 연동이 필요한 확장 단계에서 도입한다.

#### 도입 예정 시점 및 이유

- **IR 실시간 주가 API 연동**: 클라이언트에서 주기적 refetch가 필요한 경우
- **뉴스 CMS 연동**: 클라이언트 캐싱, 낙관적 업데이트가 필요한 경우
- **서버 상태 분리**: 캐싱·로딩·에러 상태를 일관되게 관리할 페이지가 2개 이상 생기는 경우

#### 대안과의 비교

| 기준 | **TanStack Query v5** | SWR | Redux Toolkit Query | Zustand + fetch |
|------|:--------------------:|:---:|:-------------------:|:---------------:|
| 캐싱 자동화 | ✅ | ✅ | ✅ | 직접 구현 |
| SSR/prefetch 지원 | ✅ (공식) | ✅ | ✅ | 직접 구현 |
| Mutation 처리 | ✅ (풍부) | 제한적 | ✅ | 직접 구현 |
| DevTools | ✅ | 제한적 | ✅ | ❌ |
| 번들 크기 | 중간 | 작음 | 큼 | 작음 |
| 학습 비용 | 중간 | 낮음 | 높음 | 낮음 |

#### 팀 역량 적합성

- React Query(TanStack Query의 이전 이름)는 국내 팀에서 사실상 표준으로 자리잡아 경험 보유 가능성이 높다.
- 공식 문서와 한국어 자료가 풍부하다.

## 2. 백엔드

이 프로젝트는 별도의 백엔드 서버 없이 **Next.js의 서버 기능**으로 백엔드 역할을 수행한다. 회사소개 홈페이지의 특성상 복잡한 비즈니스 로직보다 데이터 전달과 폼 처리가 주된 서버 역할이다.

### 2-1. Next.js Server Actions

#### 선택 이유

- **문의 폼 처리**: `/contact` 페이지의 문의 제출을 API Route 없이 서버 함수로 직접 처리한다.
- **타입 공유**: 클라이언트와 서버가 같은 TypeScript 타입을 공유하여 별도 API 스키마 정의가 불필요하다.
- **점진적 향상**: JavaScript가 비활성화된 환경에서도 HTML `<form>` 기본 동작으로 폴백된다.
- **코드 간결화**: 별도 API 엔드포인트 파일 없이 컴포넌트 파일과 가까운 위치에 서버 로직을 배치한다.

#### 대안과의 비교

| 기준 | **Server Actions** | API Routes (`/api/*`) | 외부 백엔드 (Express, NestJS) |
|------|:-----------------:|:---------------------:|:-----------------------------:|
| 설정 복잡도 | 낮음 | 낮음 | 높음 |
| 타입 공유 | ✅ (자동) | 수동 | 별도 스키마 필요 |
| 단순 폼 처리 | ✅ (최적) | 가능 | 과도함 |
| REST API 노출 | ❌ | ✅ | ✅ |
| 외부 시스템 연동 | 제한적 | ✅ | ✅ |

- **API Routes 병행 사용**: 외부 서비스(이메일 발송 등)와 연동이 필요한 경우 `src/app/api/` 경로의 Route Handlers를 사용한다.
- **외부 백엔드 미채택 이유**: 회사소개 홈페이지 규모에서 별도 백엔드 서버를 운영하는 것은 인프라 비용과 운영 부담이 과도하다.

#### 팀 역량 적합성

- Next.js를 사용하는 팀이라면 추가 학습 없이 사용 가능하다.
- React 19의 `useActionState`와 함께 사용 시 폼 상태 관리 코드가 크게 단순해진다.

## 3. DB / 스토리지

### 3-1. MVP 단계: DB 없음

#### 근거

MVP의 대부분 콘텐츠는 정적이다.

| 콘텐츠 | 데이터 소스 | DB 필요 여부 |
|--------|-----------|------------|
| 회사 소개 텍스트 | 코드 내 상수 또는 JSON | ❌ |
| 사업분야 설명 | 코드 내 상수 또는 JSON | ❌ |
| IR 재무정보 | 정적 JSON 파일 | ❌ |
| 공시 목록 | 정적 JSON 또는 DART 링크 | ❌ |
| 뉴스 목록 | 정적 JSON 파일 | ❌ |
| 문의 폼 제출 | 이메일 발송 (수신만) | ❌ |

콘텐츠를 `src/lib/data/` 폴더의 TypeScript 상수 또는 JSON 파일로 관리하면, DB 없이도 타입 안전하게 데이터를 사용할 수 있다.

### 3-2. MVP 이후 확장 시 검토 옵션

#### 옵션 A: Headless CMS (뉴스/공시 콘텐츠 관리용)

| CMS | 특징 | 추천 상황 |
|-----|------|----------|
| **Sanity** | 실시간 협업, 풍부한 쿼리 언어(GROQ) | 뉴스 콘텐츠 빈번히 업데이트 시 |
| Contentful | 엔터프라이즈 표준, 비용 높음 | 대규모 콘텐츠 팀 운영 시 |
| Notion + Notion API | 비개발자 친화적 | 팀이 이미 Notion 사용 중일 때 |

#### 옵션 B: Supabase (문의 내역 저장/관리용)

- PostgreSQL 기반 BaaS(Backend as a Service)
- 문의 내역 어드민 관리가 필요해질 때 도입
- Next.js와의 통합이 잘 지원됨
- 무료 티어로 소규모 운영 가능

#### 옵션 C: 정적 파일 유지 (현재)

뉴스, IR 데이터 업데이트 빈도가 낮다면, PR로 JSON 파일을 직접 수정하고 Vercel에서 자동 재배포하는 방식이 운영 비용 없이 가장 단순하다.

## 4. 인프라

### 4-1. Vercel

#### 선택 이유

- **Next.js 네이티브 지원**: Next.js 제작사의 배포 플랫폼으로, App Router, Server Actions, Edge Runtime을 제로 설정으로 지원한다.
- **자동 배포**: GitHub `main` 브랜치 push 시 자동 프로덕션 배포가 트리거된다.
- **Preview 배포**: PR마다 별도 URL의 Preview 환경이 자동 생성되어 코드 리뷰 시 실제 동작을 확인할 수 있다.
- **내장 CDN**: 정적 자산이 글로벌 Edge 네트워크에서 서빙되어 LCP 개선에 유리하다.
- **Analytics**: Vercel Analytics로 Core Web Vitals를 대시보드에서 모니터링할 수 있다.
- **무료 티어**: 개인/소규모 프로젝트에서는 Hobby 플랜으로 충분하며, 트래픽 증가 시 Pro로 업그레이드 가능하다.

#### 대안과의 비교

| 기준 | **Vercel** | Netlify | AWS Amplify | 자체 서버 (EC2/NCP) |
|------|:---------:|:-------:|:-----------:|:------------------:|
| Next.js 최적화 | ✅ (최고) | 보통 | 보통 | 직접 설정 |
| 설정 없는 배포 | ✅ | ✅ | 보통 | ❌ |
| Preview 배포 | ✅ | ✅ | ✅ | ❌ |
| 비용 (소규모) | 무료 가능 | 무료 가능 | 무료 가능 | 서버 비용 발생 |
| SSR 지원 | ✅ (완벽) | ✅ | 보통 | 직접 구현 |
| 국내 접속 속도 | 보통 | 보통 | ✅ (서울 리전) | ✅ (서울 리전) |

- **AWS Amplify 미채택 이유**: Next.js 지원이 Vercel 대비 제한적이며 설정이 복잡하다. Next.js 15 App Router 기능 일부가 지원되지 않을 수 있다.
- **자체 서버 미채택 이유**: 회사소개 홈페이지 규모에서 서버 운영, 보안 패치, 스케일링을 직접 관리하는 것은 운영 비용 대비 효용이 낮다.
- **Netlify 미채택 이유**: 기능적으로 유사하지만 Next.js 통합에서 Vercel이 더 안정적이다. Next.js 최신 기능(Server Actions 등)의 지원 시점이 Vercel보다 늦을 수 있다.

#### 한계 및 대응

- **국내 접속 속도**: Vercel의 국내 PoP(Point of Presence)은 서울 리전을 가진 AWS 대비 체감 속도가 떨어질 수 있다. 다음 방법으로 보완한다.
  - 정적 자산은 Vercel Edge CDN으로 서빙하되, `next/image`의 WebP 변환 + 지연 로딩을 적용해 이미지 전송량을 최소화한다.
  - ISR(Incremental Static Regeneration)을 활용해 캐시 히트율을 높인다.
  - 추후 트래픽 분석 후 필요 시 CloudFront(서울 리전) 또는 CloudFlare를 CDN 앞단에 추가 검토한다.

#### 팀 역량 적합성

- GitHub 계정 연동만으로 배포가 완료되어 별도 인프라 지식 없이 사용 가능하다.
- Preview URL을 통해 비개발자(기획, 디자인)도 결과물을 바로 확인할 수 있다.

---

### 4-2. GitHub Actions (CI)

#### 선택 이유

- **자동 품질 검사**: PR 생성 시 ESLint, TypeScript 타입 체크, 빌드 검증을 자동 실행한다.
- **Vercel 연동**: Vercel의 GitHub 앱과 연동 시 Preview 배포 상태를 PR 체크로 확인할 수 있다.
- **무료**: 공개 레포 및 일정 시간 이하 사용 시 무료다.

#### 주요 워크플로

```
PR 생성 → [ESLint 검사] + [tsc --noEmit] + [pnpm build] + [Vitest] + [Lighthouse CI]
        → 모두 통과 시 Merge 허용
```

- **Lighthouse CI**: `@lhci/cli`를 GitHub Actions에 추가하여 PR마다 Lighthouse 점수를 코멘트로 표시한다. 성능 점수 80점 미만 시 빌드 실패 처리하여 요구사항의 Must Have 기준을 자동으로 검증한다.
- **Vitest**: 단위 테스트 실행 (5-4 테스트 전략 참고)

---

### 4-3. pnpm

#### 선택 이유

- **디스크 효율**: 패키지를 전역 스토어에 한 번만 저장하고 하드링크로 참조하여 `node_modules` 크기가 크게 줄어든다.
- **속도**: npm, yarn 대비 install 속도가 빠르다.
- **엄격한 의존성**: 실제로 선언된 패키지만 접근 가능하여 의도치 않은 패키지 사용을 방지한다.
- **Monorepo 지원**: 향후 다중 패키지 구조로 확장할 때 workspace 기능을 기본 제공한다.

#### 대안과의 비교

| 기준 | **pnpm** | npm | yarn (v1) | yarn Berry (v2+) |
|------|:-------:|:---:|:---------:|:----------------:|
| 설치 속도 | ✅ (가장 빠름) | 느림 | 보통 | 빠름 |
| 디스크 사용량 | ✅ (최소) | 많음 | 많음 | 보통 |
| 의존성 엄격성 | ✅ | ❌ | ❌ | ✅ |
| Next.js 공식 지원 | ✅ | ✅ | ✅ | ✅ |
| 팀 친숙도 | 높아지는 중 | 높음 | 높음 | 낮음 |

- **npm 미채택 이유**: 속도와 디스크 효율이 떨어진다. `CLAUDE.md`에서 npm 사용 금지를 명시.
- **yarn Berry 미채택 이유**: Plug'n'Play 방식이 일부 도구와 호환 문제를 일으킬 수 있다.

- **IDE 연동**: VSCode `editor.formatOnSave: true` 설정으로 저장 시 자동 포맷을 적용한다.
- **CI 강제**: GitHub Actions에서 `eslint --max-warnings 0`으로 경고도 빌드 실패 처리한다.

### 5-2. 폼 처리: react-hook-form + zod

#### 선택 이유

- **react-hook-form**: 비제어 컴포넌트 기반으로 리렌더링을 최소화하여 폼 성능이 뛰어나다. React 19의 `useActionState`와 통합하여 Server Actions 폼 상태 관리를 단순화한다.
- **zod**: TypeScript-first 스키마 검증 라이브러리. 동일한 스키마를 클라이언트(유효성 검사)와 서버(Server Action 입력 파싱) 양쪽에서 재사용하여 중복 코드를 제거한다.

#### 예시 패턴

```typescript
// src/lib/schemas/contact.ts
import { z } from 'zod';

export const contactSchema = z.object({
  name: z.string().min(1, '이름을 입력해 주세요'),
  email: z.string().email('유효한 이메일을 입력해 주세요'),
  type: z.enum(['사업제휴', '채용', '언론', '기타']),
  title: z.string().min(1, '제목을 입력해 주세요'),
  content: z.string().min(10, '내용을 10자 이상 입력해 주세요'),
});
// → 클라이언트 react-hook-form resolver + 서버 Server Action 양쪽에서 재사용
```

### 5-3. 애니메이션 전략

#### MVP: CSS + Intersection Observer API

- 스크롤 트리거 페이드인·슬라이드인은 `IntersectionObserver`를 `useScrollAnimation` 커스텀 훅으로 구현하고, SCSS `@keyframes`로 애니메이션을 정의한다.
- 수치 카운트업은 `requestAnimationFrame` 기반의 경량 유틸리티 함수로 구현한다.
- 별도 애니메이션 라이브러리 없이 번들 크기 증가를 억제한다.

#### Should Have 단계: Framer Motion 검토

- 복잡한 레이아웃 전환, 드래그 인터랙션이 필요해지는 경우 Framer Motion 도입을 검토한다.
- 도입 기준: `IntersectionObserver`로 구현하기 어려운 인터랙션이 2개 이상 필요해질 때.

### 5-4. 테스트 전략

#### MVP 최소 범위

| 도구 | 역할 | 범위 |
|------|------|------|
| **Vitest** | 단위 테스트 | 유틸리티 함수, zod 스키마 검증, 데이터 변환 |
| **Playwright** | E2E 스모크 테스트 | 메인·사업분야·문의하기 페이지 기본 렌더링 + 폼 제출 플로우 |

#### 방침

- 컴포넌트 단위 테스트는 MVP 범위에서 제외한다 (정적 콘텐츠 위주로 ROI가 낮음).
- Vitest는 GitHub Actions CI에 포함하여 PR마다 자동 실행한다.
- Playwright E2E는 배포 후 스모크 검증용으로 선택적 실행한다.
