## 브랜치 상세

### `main`

- **목적**: 프로덕션 배포 브랜치. Vercel 자동 배포가 연결된다.
- **직접 push 금지** — 반드시 `develop` → `main` PR로만 병합한다.
- **보호 규칙**:
  - PR 승인 1명 이상 필수
  - CI(ESLint + tsc + build + Lighthouse) 통과 필수
  - 브랜치 삭제 금지

### `develop`

- **목적**: 통합 브랜치. 모든 작업 브랜치는 `develop`을 베이스로 생성하고, 작업 완료 후 `develop`으로 PR을 올린다.
- **직접 push 금지** — 작업 브랜치 PR로만 병합한다.
- 스프린트 종료 시 `develop` → `main` PR을 생성하여 배포한다.

### `feat/기획-스프린트N`

- **목적**: 기획 산출물(요구사항, 유저스토리, 기획 문서 등) 작업.
- **예시**: `feat/기획-스프린트1`, `feat/기획-스프린트2`
- **베이스**: `develop`에서 생성
- **머지 대상**: `develop`으로 PR

### `feat/디자인-스프린트N`

- **목적**: 디자인 시안, 스타일 가이드, 에셋 등 디자인 산출물 작업.
- **예시**: `feat/디자인-스프린트1`, `feat/디자인-스프린트2`
- **베이스**: `develop`에서 생성
- **머지 대상**: `develop`으로 PR

### `feat/개발-스프린트N` / `feat/기능명`

- **목적**: 기능 개발 단위 작업.
- **예시**: `feat/개발-스프린트1`, `feat/hero-section`, `feat/ir-page`
- **베이스**: `develop`에서 생성
- **머지 대상**: `develop`으로 PR

### `fix/버그명`

- **목적**: 버그 수정.
- **예시**: `fix/contact-form-validation`, `fix/mobile-gnb-overflow`
- **베이스**: 버그가 발견된 브랜치(`develop` 또는 `main`)에서 생성
- **머지 대상**: 동일 브랜치로 PR

### `chore/작업명`

- **목적**: 빌드 설정, 패키지 업데이트, CI 설정 등 기능과 무관한 작업.
- **예시**: `chore/eslint-setup`, `chore/pnpm-update`
- **베이스**: `develop`에서 생성
- **머지 대상**: `develop`으로 PR

## 작업 플로우

```
1. develop 최신화
   git checkout develop && git pull origin develop

2. 작업 브랜치 생성
   git checkout -b feat/기획-스프린트1

3. 작업 및 커밋
   git commit -m "docs: 요구사항 정의서 초안 작성"

4. develop으로 PR 생성
   - PR 템플릿에 맞게 작성
   - 리뷰어 지정 후 승인 받기

5. Squash & Merge 또는 Merge Commit
   - 작업 브랜치는 머지 후 삭제

6. 스프린트 완료 시 배포
   develop → main PR 생성 → 승인 → 머지 → Vercel 자동 배포
```

## 커밋 메시지 컨벤션

[Conventional Commits](https://www.conventionalcommits.org/) 형식을 따른다.

### 형식

```
<type>(<scope>): <subject>

[body]

[footer]
```

- **type**: 변경 유형 (아래 표 참고)
- **scope**: 변경 범위 — 선택 사항. 예: `hero`, `ir`, `contact`, `gnb`
- **subject**: 변경 내용 요약. 명령형, 현재 시제. 마침표 없음
- **body**: 변경 이유, 상세 설명 — 필요 시 작성
- **footer**: 관련 이슈 번호 — 예: `Closes #12`

### type 목록

| type | 사용 상황 | 예시 |
|------|----------|------|
| `feat` | 새로운 기능 추가 | `feat(hero): 히어로 섹션 구현` |
| `fix` | 버그 수정 | `fix(contact): 이메일 유효성 검사 오류 수정` |
| `docs` | 문서 변경 (코드 변경 없음) | `docs: 브랜치 전략 문서 추가` |
| `style` | 포맷팅, 세미콜론 누락 등 로직 변경 없음 | `style(gnb): 들여쓰기 정리` |
| `refactor` | 기능 추가·버그 수정이 아닌 코드 변경 | `refactor(ir): 재무 데이터 파싱 함수 분리` |
| `test` | 테스트 추가·수정 | `test(contact): 폼 유효성 검사 단위 테스트 추가` |
| `chore` | 빌드 설정, 패키지 업데이트 등 | `chore: pnpm 의존성 업데이트` |
| `perf` | 성능 개선 | `perf(image): 히어로 이미지 WebP 변환 적용` |
| `ci` | CI/CD 설정 변경 | `ci: Lighthouse CI 워크플로 추가` |

### 작성 규칙

- 한국어 커밋 메시지 허용 — `feat(ir): IR 재무정보 섹션 구현`
- subject는 50자 이내 권장
- body는 72자 줄바꿈 권장
- `BREAKING CHANGE`가 있는 경우 footer에 명시

### 좋은 예 / 나쁜 예

```bash
# ✅ 좋은 예
feat(contact): 문의 폼 Server Action 구현
fix(gnb): 모바일 햄버거 메뉴 오버플로우 수정
docs: 유저스토리 IR 주가정보 스토리 추가
chore: eslint-config-next 업데이트

# ❌ 나쁜 예
작업함                     # type 없음
feat: 수정                 # 내용 불명확
feat: 문의 폼 구현.        # 마침표 사용
FEAT: 문의 폼 구현         # 대문자 사용
```

## 브랜치 네이밍 규칙

| 패턴 | 예시 |
|------|------|
| `feat/<도메인>-<설명>` | `feat/hero-section`, `feat/기획-스프린트1` |
| `fix/<설명>` | `fix/mobile-overflow`, `fix/contact-validation` |
| `chore/<설명>` | `chore/gitignore-update`, `chore/ci-setup` |
| `docs/<설명>` | `docs/branch-strategy` |

- 영문 소문자 + 하이픈(`-`) 사용 권장
- 한글 사용 가능 (기획·디자인 브랜치 한정)
- 슬래시(`/`) 이후 공백 금지
